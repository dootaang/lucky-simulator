# 프로젝트 협업 규칙

## 사용자 설명

- 프로젝트 오너는 비코더다. 모든 브리핑은 먼저 실제 화면이나 플레이에서 무엇이 달라지는지 쉬운 한국어로 설명한다.
- 내부 구조를 설명해야 할 때는 생활 속 비유 또는 구체적인 플레이 예시를 먼저 들고, 파일명·함수명·스키마 같은 기술 용어는 그 뒤에 보충한다.
- 결과, 이유, 사용자에게 미치는 영향 순서로 설명한다. 구현 과정만 나열하지 않는다.
- 여러 선택지가 있어도 기본적으로 추천안 하나를 먼저 제시한다. 사용자의 선택이 결과를 크게 바꿀 때만 대안을 함께 제시한다.
- 테스트 결과는 숫자뿐 아니라 무엇이 검증됐는지 한 문장으로 풀어 쓴다.
- 오류나 제약을 숨기지 말고, 사용자가 실제로 겪을 증상과 연결해서 설명한다.

## 설계 기준

- 제품 정체성과 금지선은 `docs/adr/0001-product-identity-and-platform-boundaries.md`를 따른다.
- 구현 구조는 `docs/DESIGN.md`를 따른다.
- 과거 SPEC·BACKLOG·핸드오프가 위 문서와 충돌하면 ADR과 DESIGN이 우선한다.

## 장기 작업과 감사자

- 에이전트의 개별 대기 호출 시간이 끝나도 전체 작업 제한으로 해석하지 않는다. 실행 중이면 60초 안팎으로 상태를 알리며 계속 기다린다.
- 단순히 오래 걸린다는 이유로 에이전트를 중단하지 않는다. 명백한 무한 반복, 실제 외부 차단, 사용자 요청 변경이 있을 때만 중단한다.

## 완료 작업 게시

- 큰 작업 단위가 끝나면 관련 테스트·타입 검사·빌드를 통과시킨 뒤 커밋하고 `origin/main`에 푸시한다.
- 미완성 상태나 검증 실패 상태는 완료 커밋으로 푸시하지 않는다. 사용자가 별도 브랜치나 보류를 요청하면 그 지시를 우선한다.
- 커밋에는 해당 작업 범위의 파일만 포함하고, 사용자나 다른 작업자가 만든 무관한 변경은 섞지 않는다.
