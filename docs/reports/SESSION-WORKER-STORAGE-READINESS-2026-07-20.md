# 세션 Worker·저장 엔진 준비 감사 (2026-07-20)

## 사용자에게 보이는 결론

- 지금 앱에 새 저장 엔진을 설치하거나 세션 전체를 Worker로 옮기지 않는다.
- 먼저 모든 봇의 버튼이 같은 안전한 행동 통로를 지나도록 정리한다.
- 현재 빠른 소녀전선의 플레이 결과·세이브·RNG는 바꾸지 않는다.
- 두 번째 대형 봇이나 Firebase 동기화에서 실제 병목이 측정될 때 준비된 경계 뒤쪽만 교체한다.

## 이미 갖춘 기반

| 기반 | 현재 구현 | 판정 |
|---|---|---|
| 카드 실행 격리 | 세션별 `card-runtime.worker.ts`, revision/requestId 검증, 늦은 응답 폐기 | 재사용 |
| 브라우저 저장 | 공식 SQLite WASM `OpfsDb`를 전용 Worker에서 사용, 실패 시 IndexedDB 폴백 | 유지 |
| 행동 내구성 | 작은 action WAL을 먼저 기록하고 전체 저장은 배경 처리 | 유지 |
| 결정론 | 엔진 저널, 상태 해시, RNG 위치, 리플레이·에폭 검증 | 불변 |
| 화면 행동 통로 | `SimulationActionHandler`가 GFL·여관·전투·선언형 화면을 한 곳으로 연결 | 공용 계약으로 승격 |
| 대화 성능 | 보이는 메시지 창문 렌더, 에셋 LRU, revision 분리, 구형 저장 압축 | 유지 |

## 아직 남은 직접 결합

1. 화면 전용 `simulation-action.ts`가 세션의 trace 타입을 중복 선언한다.
2. 행동 요청에 세션 ID와 단조 증가 행동 순번이 없어 미래 Worker의 늦은 결과를 구조적으로 구분하기 어렵다.
3. `PlayerPage`가 ledger/narrated/scene 분기와 직렬 큐를 직접 소유한다.
4. GFL 계기판·결정 카드·HUD 등 일부 소비자는 `runtime.select()`와 raw state를 화면에서 직접 읽는다.
5. 일반 채팅 send/continue, undo/redo, 설정 변경은 시뮬레이션 행동 통로 밖에 있다.
6. Firebase 동기화를 위한 기기·부모 상태 식별 계약은 아직 제품 계약이 아니다.

## 구현 경계

### 이번 작업에서 한다

- 공용 행동 요청·결과·순번 계약을 `@simbot/session`에 둔다.
- 기존 UI 행동 핸들러를 새 계약의 얇은 어댑터로 바꾼다.
- 세션 교체 시 이전 큐의 늦은 결과가 새 화면에 적용되지 않는 테스트를 추가한다.
- 화면에 필요한 selector를 묶어 읽을 수 있는 범용 view bundle 계약과 캐시를 만든다.
- 모듈 카탈로그의 두 번째 대형 카나리아로 공용 event/selector 경계를 검증한다.

### 이번 작업에서 하지 않는다

- PlaySession 정본 전체 Worker 이전
- SQLite/OPFS를 다른 VFS나 IndexedDB로 강제 교체
- Firebase Auth·Firestore·Storage 연결
- 대형 에셋 원본의 클라우드 업로드
- 세이브 스키마와 GFL 밸런스 변경

## 전환 안전장치

- 기존 `PlaySession`은 인프로세스 테스트가 가능해야 한다.
- 행동 전후 저널·RNG·stateHash·로그가 기존 경로와 바이트 동일해야 한다.
- Worker 시제품은 전체 상태를 왕복시키지 않고 화면 묶음만 반환해야 한다.
- 저장소 후보는 인터넷 벤치마크가 아니라 오너 장기 세션의 실제 읽기·쓰기 경쟁으로만 채택한다.
- Firebase 동기화는 로컬 우선이며, 같은 부모에서 갈라진 진행은 덮어쓰지 않고 분기 세이브로 남긴다.
