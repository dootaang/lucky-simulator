# ADR 0002: 럭키 시뮬레이터 — Risu 호환 장기 시뮬 플레이어

- 상태: 승인
- 결정일: 2026-07-12
- 적용 범위: 제품 정체성, 소비자 경험, 호환 계층, 엔진, 편집기
- 상위 결정: ADR 0001을 **갱신**(대체가 아니라 초점 이동 — 아래 참조)

## 맥락

두 번의 독립 펜타 브레인스토밍(이 대화의 지휘자 세션 + 별도 터미널의 Codex 세션)과 세 단원(Codex·Gemini Pro·Flash)이
**서로 보지 않고 같은 결론에 수렴**했다. 또한 실제 인기 RisuAI 시뮬봇 13장(용사여관·Alternate Hunters V2·소녀전선·
Belladonna·Merry Sisters·THE MORTAL/DOMINIUM REALM·Tesselia·Isekai Re·루미나·오렌티아·Rote 등)을 분해·정독했다.

핵심 발견 — 리스 시뮬봇의 "게임 엔진"이 어디에 사는가:

1. 시뮬 로직은 카드의 `module.risum` 안 **triggerlua(Lua) + CBS 매크로 + 정규식 + `backgroundHTML` 상태패널 +
   `defaultVariables` 초기상태**에 흩어져 있다. 큰 카드는 Lua만 300k자(용사여관 302k·Belladonna 327k·Merry 367k).
2. 인코딩 방식이 **세 갈래**로 뚜렷하다:
   - **Lua 태그 파서**(용사여관): LLM이 `[ysp_gold::+40::사유]` 같은 태그를 뱉으면 Lua가 파싱→변수 조작.
   - **CBS·정규식**(DOMINIUM·Isekai Re): Lua 없이 LLM이 상태창을 텍스트로 출력, 정규식이 매핑. 로직이 diffuse.
   - **순수 로어북**(Tesselia 353항목·루미나): 세계관 중심, 기계 상태 거의 없음 — 엔진이 필요 없다.
3. **결정적 통찰**: 용사여관의 Lua는 `reindexYspState`(1번 메시지부터 전체 재파싱 = 수동 이벤트 소싱)와
   `bk_turn_key`(리롤·턴취소 시 상태 복원 = 수동 롤백 백업)를 손으로 구현한다. **리스가 상태·롤백·재인덱싱을
   안 주기 때문에 제작자가 300k Lua로 직접 짜는 것이다.**

즉 문제의식이 선명해졌다: 리스에서 시뮬봇은 (a) 상태가 컨텍스트 누적으로 왜곡되고, (b) 스와이프·리롤 시 글만 바뀌고
누적 사실이 꼬이며, (c) 제작자가 이 모든 걸 방대한 Lua로 손수 방어해야 한다. 우리 엔진은 이 셋을 **네이티브로** 해결한다.

## 결정 — 제품 정체성

> **럭키 시뮬레이터(Lucky Simulator)** — 당신의 Risu 봇카드·프롬프트·로어·페르소나가 익숙한 방식 그대로 돌아가는데,
> 밑에서 결정론 엔진이 상태·판정·재화를 소유해 수백 턴을 굴려도 무너지지 않는 곳.

공개 문구(대외 약속): *"Risu에서 쓰던 그대로. 수백 턴 뒤에도 무너지지 않는 시뮬레이션."* / *"Risu에 익숙한 사용 환경과
주요 콘텐츠 호환."* — "Risu와 동일"이라고 약속하지 않는다(아래 §금지선). "럭키 리스"는 커뮤니티 별칭으로만.

### ADR 0001과의 관계

ADR 0001("노코드 시뮬 메이커 + 플레이어")의 **엔진·아키텍처 결정(장르 중립 커널·모듈·SimPack·화면 선언)은 전부 유효**하다.
바뀌는 것은 **제품의 얼굴과 우선순위**다: "노코드 게임 제작기"가 아니라 **"Risu 호환 장기 시뮬 플레이어"**가 사용자에게
보이는 정체성이다. 제작기는 필요한 내부 기능이되 첫 화면이 아니다. **소비자 플레이 경험을 먼저 완성한다.**

## 제품 헌법 (후속 구현보다 우선 — 최상위 3문장)

1. 럭키 시뮬레이터는 **Risu 사용자가 익숙한 방식으로 시뮬봇을 장기간 플레이하는 환경**이다.
2. 성공은 **Risu 기능 개수가 아니라, 실제 카드가 수백 턴 동안 무너지지 않는 것**으로 측정한다(디싱크 0 하버스로 상시 검증).
3. 호환을 넓히기 전에 **매 단계에서 엔진·서사 디싱크 0을 *기계적으로* 증명**하고, 롤백·근거 기억은 일반 채팅에까지 보편 적용한다.

이하 세부 원칙:

4. **엔진이 사실을 소유한다.** 돈·시간·위치·호감·스탯·인벤토리·장비·퀘스트·시설·전투·생존 상태는 엔진만 바꾼다.
   LLM은 이것들을 서술할 수 있으나 직접 변경할 수 없다.
5. **실제 선택지는 엔진만 제시한다.** LLM 본문은 상황 설명, 선택지는 엔진 결정 카드가 권위 원본.
6. **되돌리면 상태도 함께 되돌아간다.** 스와이프·재생성·수정은 **턴 체크포인트**에서 파생되며, 글만 바뀌고
   골드·퀘스트·기억이 이전 결과로 남으면 안 된다. (리스 카드가 `bk_turn_key`로 손수 하던 것을 엔진이 네이티브로.)
7. **미지원 기능은 삭제하지 않고 보존·경고한다.** 완전 호환을 과장하지 않고 **호환성 여권**으로 등급을 정직히 표시한다.
8. **카드별 앱 코드를 추가하지 않는다.** 새 카드의 비용은 모듈·데이터 조합으로.
9. **엔진 UI는 카드가 필요한 만큼만 나타난다**(§3모드). 일반 카드는 리스형 채팅으로 즉시.
10. **안전한 기본값이 자유도보다 먼저다.** 임의 Lua/DOM/무제한 플러그인 실행은 금지, 보존·정적분석만.

## 아키텍처 — 3모드는 별개 제품이 아니라 '설치 모듈 개수'다

| 모드 | 대상 카드(실측) | 엔진 개입 | 내부 구현 |
|---|---|---|---|
| 일반 채팅 | Tesselia·루미나·오렌티아(순수 로어) | 없음 | 엔진 모듈 0개 |
| 엔진 보조 | 일부 상태만 | 시간·관계·기억 선택 적용 | 일부 모듈 |
| 완전 시뮬 | 용사여관·얼헌(Lua/CBS 시뮬) | 경제·퀘스트·전투·시간 전체 | 핵심 장르 모듈 |

한 세션이 **설치된 모듈과 엔진 소유 상태 범위에 따라 자연스럽게 달라진다.** 세 제품으로 갈라 구현하지 않는다.
중요: **롤백·턴 체크포인트·근거 기억은 일반 채팅 모드에도 적용**된다(모듈 0개여도 "대화+기억"을 되감는 건 리스가
깔끔히 못 하는 보편 무기). 즉 엔진 없는 모드조차 럭키를 쓸 이유가 있다.

## Risu 시뮬 로직 번역 정책 (B안 재확인 — 실측 기반)

우리는 리스의 Lua/CBS/정규식을 **실행하지 않고 의미를 우리 엔진으로 번역**한다. 인코딩 방식별 전략:

- **Lua 태그 파서** → **태그를 엔진 이벤트로 직역.** `[ysp_gold::+40]` → `gold_delta`, `[ysp_checkin::방::NPC::박]` →
  `checkin`, `[ysp_nextday]` → `day_end`(자동 체크아웃·인건비 정산 포함). 제작자가 손코딩하던 누적·롤백·재인덱싱·정산
  루틴은 **엔진이 대신** 한다(우리 sessionJournal·모듈이 곧 그들의 reindexYspState·bk_turn_key의 네이티브 대체).
- **CBS·정규식** → LLM 상태창 텍스트를 **전처리 파서**로 흡수해 엔진 상태로, 본문에서 제거 후 자체 위젯으로 표시.
  `/ghost` 같은 인풋 명령은 정규식 트리거로 가로챔.
- **순수 로어북** → **일반 채팅 폴백.** 로어북(키/순서/재귀/예산)을 컨텍스트 주입기로 로드, `[<img="…">]`는 에셋 위젯으로.

### 어려운 경우 — 정직히 명시

- **보조 LLM(axLLM) 추론 상태**(얼헌 상태창·Merry Sisters 퀘스트/날씨): 리스는 상태를 *수학*이 아니라 *두 번째 LLM 추론*으로
  갱신한다. 100% 결정론화 시 "본문엔 임무 완료, 상태창은 진행 중" 부조화 위험. → **이벤트를 본문 태그로 뱉게 유도 + 우리
  엔진 이벤트로 매핑**하거나, 역추론 판정 유닛·수동 오버라이드 폴백을 둔다. Merry Sisters류(3중 axLLM + HTML 다이스 버튼)는
  "불가" 등급 — 보존·경고 후 룰 재기획 대상.
- **HTML/CBS 인터랙티브**(버튼 콜백·다이스): 안전 위젯으로 재현 가능한 것만, 임의 DOM·JS는 금지.

## 호환성 여권 (대외 신뢰의 핵심)

카드·모듈 임포트 시 항목별 4등급을 표시: **그대로 호환 / 안전하게 변환 / 원본만 보존 / 현재 미지원.**
"호환됩니다"라는 뭉뚱그린 약속 대신 카드마다 정직한 진단을 준다. (기존 Card MRI·Risu 호환 봉투가 이 자리.)

## MVP 합격 기준 (북극성 = 기능 개수 아님)

> **실제 Risu 시뮬봇 한 장이 300턴 동안 상태 불일치 없이 작동하는가?**

상시 측정하는 디싱크 하버스 지표(전부 0 목표): 엔진·서사 상태 불일치 / 가짜 선택지 / 미허용 상태 변경 /
재생성·롤백 후 잔여 상태 / 저장·재개 후 상태 차이 / 카드별 전용 앱 코드 / 미지원 Risu 데이터의 조용한 손실.

### 카나리아 3장 (실측으로 선정)

1. **용사여관** — 가장 고도화된 Lua 태그 파서. 이게 첫 완결 플레이까지 버그 없이 돌면 코어 아키텍처 대부분 검증.
2. **THE DOMINIUM REALM** — Lua 없이 정규식·CBS 상태창 + 인풋 명령(`/ghost`). *다른 인코딩*을 증명(태그 파서에만 안 맞춤).
3. **The Chronicles of Tesselia** — 353 로어·게임수학 없음. **엔진 없는 일반 채팅 모드**가 굴러가는지 증명.

## 금지선 (MVP 합격 전)

- 대외 "Risu와 동일" 약속(패리티 경쟁 유발). 내부 목표는 높은 호환이되 약속은 여권으로 증명.
- 임의 Lua/DOM/JS 실행, 무제한 플러그인, TTS·이미지생성·번역·호드 등 범용 RP 표면 추격.
- 카드별 플레이뷰 분기, LLM 출력의 무검수 자동 승인, 결정 카드 밖 가짜 선택지.

## 후속 순서 (요약 — 상세는 첫 수직 단면 SPEC)

세로줄 우선: 리스 패리티를 넓게 먼저 만들지 않고, **P0(최소 이사 환경) + 얇은 P2(럭키 매직)를 동시**에 하나의 카드로
관통한다. 용사여관 수직 단면 → DOMINIUM → Tesselia 순으로 카나리아. 두 번째 카드에서 앱 코드를 고쳐야 하면 구조가
아직 범용적이지 않은 것이다.

## 결과

- 리스 생태계의 이사 비용을 0에 가깝게 낮추면서, 리스가 구조적으로 못 하는 "300턴 불변성·상태 롤백"을 차별점으로 얻는다.
- 제작자가 손코딩하던 상태·롤백·재인덱싱 Lua가 **불필요**해진다 — 태그 계약만 선언하면 엔진이 준다.
- 감수: 리스 UX·콘텐츠 호환 표면(프리셋·로어·정규식·채팅 조작)을 상당량 구현해야 하며, 이것이 다음 개발의 본체다.
- 위험: 패리티 체크리스트가 엔진의 체감 가치를 삼키는 것 — 북극성(디싱크 0·한 카드 300턴)으로 매주 방어한다.
