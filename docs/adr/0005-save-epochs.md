# ADR 0005: 세이브 에폭 — 업데이트 뒤에도 같은 회차 이어가기

- 상태: 승인
- 결정일: 2026-07-18
- 적용 범위: 엔진 사건 원장, 세션 복구, 모듈 상태 이주
- 근거: `PROPOSAL-SAVE-EPOCH-2026-07-18.md` 오너 승인

## 맥락

기존 세션 복구는 저장된 모든 사건을 현재 코드로 다시 실행해 저장 당시 상태·로그·RNG와 대조했다.
같은 버전 안에서 손상이나 비결정성을 잡는 데는 강하지만, 정상적인 게임 규칙 업데이트도 과거 결과를
달라지게 만들어 저장 회차를 격리했다. 배포할 때마다 플레이어에게 새 회차를 요구할 수는 없다.

구분해야 할 것은 두 가지다. 바깥 세션 무결성 서명이 깨진 저장은 손상이다. 서명은 유효하지만 엔진
스키마 지문이 달라진 저장은 업데이트를 거친 정상 저장이다.

## 결정

1. 엔진 사건 원장을 `simbot-event-journal/0.2`로 올리고 과거를 읽기 전용 `sealedEpochs`로 보존한다.
   봉인 에폭은 현재 코드로 재실행하지 않으며, 안정 직렬화한 내용의 SHA-256 `sealHash`로 변조를 검사한다.
2. 현재 에폭은 종전처럼 전부 재실행한다. 같은 스키마 지문에서 로그·상태·RNG가 발산하면 계속 손상으로
   처리한다. 바깥 세션 `integrity` 검증도 순서와 강도를 바꾸지 않는다.
3. 업데이트를 감지하면 무결성으로 검증된 저장 시점의 head 상태를 채택하고, 등록 순서대로 모듈의
   `migrations.seal`을 적용한 뒤 새 에폭의 시작 상태로 삼는다. 이주 함수는 멱등인 순수 함수이며 RNG,
   시간, 외부 입력을 사용하지 않는다. 예외가 나면 복구 전체를 중단한다.
4. 사건 번호는 에폭을 넘어 전역으로 이어진다. 봉인 경계 이전의 `stateAt`·`truncateTo`와 시간여행은
   막되 메시지, 기억, 프롬프트 실행 기록은 보존한다. 플레이어에게 봉인 안내 메시지와 진단을 한 번 남긴다.
5. `simbot-event-journal/0.1`은 같은 지문에서 기존 재생 검증으로 읽고, 다음 저장부터 v0.2로 쓴다.
6. 카드 런타임 원장(`card-runtime-journal`)은 대상에서 제외한다. 그 원장은 엔진 사건 처리기를 다시
   실행하는 대신 이미 확정된 변수 patch를 재생하므로 코드 버전 변화로 의미가 달라지는 현재 문제가 없다.

첫 실사용 이주는 `genre.gfl`의 기존 제대 슬롯을 5칸에서 6칸으로 `null` 패딩하는 작업이다. 이미 6칸인
상태는 바꾸지 않고, 부재해도 안전한 신규 키를 억지로 주입하지 않는다.

## 결과

- 정상 업데이트 뒤에도 같은 채팅, 기억, 엔진 상태를 유지한 채 플레이를 계속한다.
- 손상 저장과 업데이트 저장의 경계가 분명해지며, 무결성·같은 버전 재생 검증은 약해지지 않는다.
- 봉인 경계 이전으로 되돌릴 수는 없다. 안내 이후 새로 쌓인 사건만 되돌릴 수 있다.
- 봉인 에폭이 누적되면 저장 크기도 늘어난다. 현재 30MB 가져오기 상한은 유지한다. 봉인 에폭 압축,
  중복 제거, 지연 로드는 사용량 측정 뒤 별도 결정한다.

## 후속 규율

상태 형태가 바뀌는 모듈 작업은 필요한 `migrations.seal`과 멱등 테스트를 함께 제공한다. 이주로 안전하게
해석할 수 없는 변경은 조용히 기본값을 확정하지 말고 예외로 복구를 중단해야 한다.
