# ADR 0001: 제품 정체성과 플랫폼 경계

- 상태: 승인
- 결정일: 2026-07-11
- 적용 범위: 제품, 엔진, 편집기, 런타임, 호환 계층
- 상위 결정: 없음

## 맥락

현재 구현은 봇카드에서 규칙을 추출해 자체 결정론 엔진으로 실행하는 데 성공했다. 그러나 엔진 상태, 행동 셀렉터, 검증기와 플레이 HUD가 용사여관의 경영·숙박·직원·의뢰 개념을 직접 알고 있다. 그 결과 Alternate Hunters V2(약칭 얼헌)처럼 다른 장르의 카드를 컴파일해도 여관형 화면과 행동 구조에 끼워진다.

이 문제는 컴파일러의 정확도만 높여서는 해결되지 않는다. 장르마다 앱 코드를 수정하는 방식은 카드 수가 늘수록 개발 공수가 선형으로 증가하며, 제작자용 도구나 공유 생태계로 확장할 수도 없다.

한편 지금까지 만든 결정론 RNG, 이벤트 적용, 판정, 전투, 의뢰, 시설, 카드·로어북·에셋 처리, 채팅 및 플레이 화면은 여전히 유효한 자산이다. 소비 경험을 폐기하고 엔진 라이브러리만 제공할 이유도 없다.

## 결정

제품을 다음과 같이 정의한다.

> 봇카드를 가져와 캐릭터·상태·규칙·화면을 시각적으로 편집하고, 결정론적 시뮬레이션으로 플레이·배포하는 LLM 네이티브 노코드 시뮬 메이커 + 플레이어 런타임.

제품은 다섯 역할을 함께 제공한다.

1. **호환 입구**: Character Card, 로어북, 프롬프트와 에셋을 가져온다.
2. **초안 생성기**: 컴파일러가 카드에서 편집 가능한 프로젝트 초안을 제안한다.
3. **제작기**: 제작자가 상태·규칙·화면을 폼과 제한된 블록으로 검수·수정한다.
4. **결정론 런타임**: 수치, 상태, 판정과 진행은 엔진이 소유하고 LLM은 서사를 담당한다.
5. **플레이어**: 완성된 프로젝트를 즉시 플레이하고, 장차 이식 가능한 패키지로 공유한다.

외부 제품 설명에서 `미니 Risu`를 정체성으로 사용하지 않는다. RisuAI 호환성과 익숙한 UX는 진입 전략이지만, 차별점은 카드에 실제 상태·규칙·게임 루프를 부여하는 결정론 시뮬레이션이다.

## 제품 헌법

다음 원칙은 후속 구현보다 우선한다.

1. **엔진이 사실을 소유한다.** LLM은 상태나 수치를 직접 확정하지 않고 검증 가능한 이벤트를 제안하며 서사를 작성한다.
2. **컴파일 결과는 초안이다.** 추출 근거, 추정, 기본 합성, 사용자 수정을 구분하고 불확실한 결정은 사용자 승인을 받는다.
3. **코어에는 장르 개념이 없다.** 여관, 객실, 직원, HP, 의뢰 같은 개념은 런타임 모듈 또는 템플릿에 속한다.
4. **화면은 명시적으로 선언한다.** 데이터 모양을 보고 HUD를 추측하지 않는다. 프로젝트의 화면 문서가 제한된 위젯과 selector를 배치한다.
5. **편집기와 플레이어는 한 제품의 두 모드다.** 제작 도구를 위해 소비 경험을 버리지 않는다.
6. **안전한 기본값이 자유도보다 먼저다.** 공유 콘텐츠는 기본적으로 데이터이며, 실행 코드는 별도 권한과 격리 경계를 가져야 한다.
7. **버전과 이식성을 첫날부터 관리한다.** 프로젝트, 모듈, 화면과 저장 데이터는 버전 및 migration 계약을 가진다.

## 아키텍처 경계

플랫폼은 다음 네 계층으로 분리한다.

### 1. 장르 중립 Kernel

상태 저장·복제, 결정론 RNG, 원자적 이벤트 실행, 이벤트 로그·replay, 저장·불러오기, 버전·migration, 권한과 검증만 소유한다.

### 2. Runtime Modules

스탯, 관계, 인벤토리, 경제, 달력, 시설, 직원, 의뢰, 전투, 숙박, 영업처럼 독립 기능을 등록한다. 각 모듈은 최소한 `id`, `version`, `dependencies`, 상태 스키마, 이벤트 처리기, selector, process, prompt facts와 migration을 제공한다.

### 3. Genre Templates

모듈 조합, 초기 데이터와 기본 화면 배치를 제공한다. 용사여관은 경영·숙박·직원·의뢰 템플릿으로, Alternate Hunters V2는 헌터 등록·등급 평가·게이트·파티·전투·정산 템플릿으로 취급한다. 템플릿은 앱 코드 포크가 아니다.

### 4. Project / SimPack

카드, 로어북, 프롬프트, 시뮬레이션 설정, 화면 선언, 에셋, 요구 모듈, 권한, 호환 버전과 출처를 담는 제작·공유 단위다.

## RisuAI 호환 및 이식 정책

적극 지원하는 범위:

- Character Card V3와 손실 없는 원본 필드 보존
- 표준 로어북과 감정·배경 에셋
- 기본 CBS 호환 계층
- 프롬프트 순서 편집, 그룹 화자, 감정 스탠딩 등 익숙한 UX 문법

선별 이식하는 범위:

- 독립적으로 검증 가능한 파서, 유틸리티와 테스트
- 현재 기술 스택에서 유지 가능한 작은 로직 또는 스타일 조각
- 원본 저장소, commit, 파일, 라이선스, 변경일과 변경 내용을 기록한 코드

초기 범위에서 제외하는 것:

- RisuAI의 Svelte UI 트리와 데이터베이스 구조 전체
- 전체 프롬프트 처리 파이프라인
- 모든 CBS 문법의 즉시 완전 호환
- 임의 Lua/JavaScript 실행
- 권한·샌드박스 없는 플러그인
- Risu 커뮤니티 카드·이미지·프롬프트의 무단 재배포

기존의 “Risu 런타임 재구현 영구 금지”는 이 결정으로 대체한다. **전면 복제는 하지 않지만, 표준 호환 계층과 출처가 관리된 선별 이식은 허용한다.**

## MVP 합격 기준

플랫폼 확장은 다음 조건으로 검증한다.

1. Alternate Hunters V2의 대표 루프 `헌터 등록 → 등급 평가 → 게이트 선택 → 파티 → 전투 → 정산`을 앱 소스 수정 없이 제작하고 플레이한다.
2. 용사여관과 Alternate Hunters V2의 HUD가 동일한 장르 껍데기로 보이지 않는다.
3. 제작자가 컴파일 결과의 출처와 불확실성을 보고 폼에서 수정할 수 있다.
4. LLM은 엔진 상태를 직접 변경하지 않고 허용된 이벤트만 제안한다.
5. 세 번째 장르 핵심 루프의 80% 이상을 기존 모듈과 위젯으로 조립한다.

Alternate Hunters V2를 만들기 위해 고정 런타임의 이벤트 분기, 셀렉터, 검증기 또는 플레이 뷰에 장르 전용 코드를 추가해야 한다면 아직 합격하지 못한 것이다.

## 금지선

MVP 합격 전에는 다음을 진행하지 않는다.

- 공개 마켓플레이스와 결제
- 범용 타일맵·월드 에디터
- 실시간 협업
- 모바일 편집기 완전판
- 임의 Lua/JavaScript 실행
- RisuAI 전체 기능 패리티
- 화면 또는 규칙의 무제한 HTML·DOM 접근
- LLM 출력의 무검수 자동 승인

## 결과

긍정적 결과:

- 용사여관 작업은 첫 공식 템플릿과 회귀 테스트로 보존된다.
- 새로운 장르의 비용이 앱 코드 수정에서 모듈·데이터 조합으로 이동한다.
- 컴파일의 불확실성이 실패가 아니라 편집기 UX로 노출된다.
- 제작과 플레이가 하나의 배포 루프로 연결된다.

감수할 비용:

- 모듈 API, 화면 선언, migration과 편집기 개발이라는 초기 기반 공수가 필요하다.
- 완전 자유 스크립팅보다 표현 범위가 제한된다.
- 기존 여관 전용 코드를 점진적으로 분리해야 한다.

## 후속 순서

1. 공통 speaker/entity/asset resolver로 스탠딩 무대 계약 수정
2. 기존 동작을 감싼 module registry facade 도입
3. 전투·스탯·인벤토리 모듈 추출
4. feature flag 기반 선언형 renderer 도입
5. Alternate Hunters V2 대표 루프 무코드 재현
6. 용사여관을 공식 템플릿으로 분리
7. 편집기 MVP
8. SimPack import/export
9. Character Card V3와 기본 CBS 호환 확대
10. 제한적 공유 갤러리와 안전한 확장 API 검토
